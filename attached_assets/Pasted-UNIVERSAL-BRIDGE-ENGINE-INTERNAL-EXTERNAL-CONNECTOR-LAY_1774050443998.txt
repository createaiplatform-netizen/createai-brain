UNIVERSAL BRIDGE ENGINE — INTERNAL ↔ EXTERNAL CONNECTOR LAYER

Goal:
Create a Universal Bridge Engine that lets my internal universe (engines, flows, dashboards, automations)
connect to ANY external system (Stripe, ads, SMS, email, marketplaces, etc.) through a single, consistent,
extensible connector layer.

Core Rules:
- Do NOT change or limit my internal engines, naming, or architecture.
- Do NOT add projections, fake data, or simulated money.
- Only create REAL connector structures that are ready to plug into real external services.
- Clearly separate INTERNAL logic from EXTERNAL connectors.
- Everything must be factual and based on the actual codebase.

Part 1 — Architecture
Design and implement a Universal Bridge Engine with:
- A central router (e.g., universalBridgeEngine.ts) that:
  - Receives internal "bridge requests" in a neutral format (e.g., { type, payload, metadata }).
  - Routes them to the correct connector (payments, ads, SMS, email, marketplace, etc.).
  - Handles success/failure responses in a consistent way.
- A shared BridgeRequest and BridgeResponse type/interface.
- A clear distinction between:
  - INTERNAL actions (what my engines want to do).
  - EXTERNAL actions (what real-world APIs will eventually do).

Part 3 — Internal Integration
- Identify key internal engines that should use the Universal Bridge Engine instead of directly simulating external behavior:
  - revenue/wealth engines
  - ad/meta/transcend engines
  - messaging/notification engines
  - marketplace/product engines
- Refactor them so that:
  - They create a BridgeRequest (e.g., { type: 'PAYMENT_CREATE_CHECKOUT', payload: {...} }).
  - They send it to the Universal Bridge Engine.
  - They handle the BridgeResponse (success, failure, NOT_CONFIGURED) gracefully.
- Do NOT reintroduce projections, fake stats, or random values.

Part 4 — Configuration & Safety
- Create a central config file (e.g., bridgeConfig.ts) that:
  - Defines which connectors are "ACTIVE" vs "NOT_CONFIGURED".
  - Reads environment variables for real API keys (but do NOT invent keys).
  - Makes it easy to turn on a connector once real credentials are added.
- If a connector is NOT_CONFIGURED:
  - The system should NOT pretend anything happened.
  - It should log clearly: which connector, which action, and that it is not yet wired to a real external service.

Part 5 — Dashboard & Visibility

  - Show status: ACTIVE or NOT_CONFIGURED.
  - Show last action attempted and its result (success, failure, NOT_CONFIGURED).
- No fake counts, no fake money, no fake campaigns.

Part 6 — No-Limits, But Real

- The Universal Bridge Engine is the ONLY place where internal → external happens.
- No attempts to simulate external money, external ads, or external infrastructure.
- Everything external must go through these connectors and must be real once configured.
