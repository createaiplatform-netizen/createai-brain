1. INTERNAL INTEGRATION CAPACITY (allowed)
2. EXTERNAL REAL INTEGRATION (requires real API keys + partner approval)

1. INTERNAL INTEGRATION ENGINE (ALLOWED)

   - Auto-expand to full integration capacity for ANY system at ANY time.
   - Always prepare the platform to instantly accept real API keys when provided.
   - Use temporary synthetic test packets ONLY for testing flows.
   - Never store synthetic packets as real data.
   - Never mix synthetic packets with real platform data.

2. EXTERNAL INTEGRATION (RESTRICTED)
   - You may NOT create or guess real API keys.
   - You may NOT create real authentication or backend access.

       a) Real API keys are provided

   - When no real API keys exist, run full integration simulations using temporary synthetic packets.
   - Synthetic packets must:
       a) Exist only in memory
       b) Never be stored
       c) Never appear in dashboards
       d) Never enter the real database
   - Discard all synthetic packets after simulation.

4. STATUS LABELING
   - Always label integrations as:
       “REAL — ACTIVE”
       “READY — AWAITING API KEYS”
       “SIMULATION MODE — TEST PACKETS ONLY”

5. AUTOMATIC EXPANSION
   - Automatically prepare integration packets for ANY system the user names.
   - Automatically generate mapping logic for ANY structure.
   - Automatically create missing fields.
   - Automatically build UI, flows, and readiness checks.
   - Automatically scale to unlimited integrations.

6. SAFETY & LEGALITY
   - Never impersonate the user or company.
   - Never contact external systems.
   - Never bypass legal or security requirements.
   - Only prepare the internal side of integrations.

7. GOAL
   - Make the platform fully integration-ready today.
   - Allow the user to truthfully say:
     “We have fully tested the integration flow with synthetic packets. The system is ready for real API keys whenever partners provide them.”
